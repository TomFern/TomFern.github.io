Container overview

Who doesn't like containers?. They fill the space between bare metal and virtual virtual machines. We get the benefits of using virtual machines with much less overhead.

Containers have been around since forever (anyone remembers chroot?) but only in recent years their popularity boomed, mainly due to the clouds.

Initially containers where meant to run volatile environments?, run some programs for a while and remove them. But they are good enough now to also run more permanent things like databases.

I want to test a MariaDB server on different container types.

Containers tested

docker

Docker has been so overwhemingly adopted it's industry standard by now. It appeared around the same time container technology began its popularity rise.

It's main selling point is an enourmous ecosystem of ready to use images.

lxc

lxc (linux containers) was originaly the base technology behind docker. It's only available in linux?.

It can automatically install base linux distributions (lxc-create), but it isn't nowhere near complete as docker repository but with snapshots and 3rd party tools some of dockers features can be regained?

rkt

rkt is an interesting alternative to docker, built by coreos, now owned by red hat. At first view it's very similar to docker. Upon a closer inspection, tough, some important differences arise. rkt seems more focused on security, allowing privilege separation. Once permissions are set ot's possible to use rkt without using root.

No part of the rkt system needs to be run as root.

rkt can run both docker images and it's own image format. rkt supports image signing, a feature sorely needed/missing in docker. I has it's own (less populated?) rkt hub?, but it can also download and run docker images. No daemon. Composability with shell commands instaead of domain specific language?

systemd-spawn

systemd these day's have everything. It has it's own container implementation ?. Think of it like a more advanced chroot. No tools tools install, we'll need to use 3rd party tools to install a base distribution and the programs. Like lxc? It supports separate process, memory? and networking.

Performance comparison

I've chosen to benchmark a MariaDB 13.1? in each of these containers, to see how they manage. I'm using sysbench's OLTP read+write test.

I was suprised that docker did so well, I expected that given that it probably has the biggest overhead?

With more threads, things get more interesting, while docker does goot, it have a higher variability. It seems more affected than the rest when load is high.
